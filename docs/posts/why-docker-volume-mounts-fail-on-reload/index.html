由 Docker 的“坑”深入理解 Linux Inode 机制

2026-06-05 · 9 分钟 · 约 4449 字

最近要在生产环境部署一个 GM 后台，本想利用 Docker 的容器的轻量化做快速部署 caddy，没想到在最基础的配置文件热重载和前端包发布上，连续踩了两个因为 Inode 变化导致的深坑。

先来看 Gemini 帮我生成的 docker compose 的配置文件。

> /opt/caddy >>>
❯ ls
Caddyfile  docker-compose.yml

version: "3.7"

services:
  caddy:
    image: caddy:2-alpine
    container_name: caddy
    restart: unless-stopped
    ports:
      - "80:80"
      - "443:443"
      - "443:443/udp" # 必须映射 UDP 端口以启用 HTTP/3 性能加速
    volumes:
      - ./Caddyfile:/etc/caddy/Caddyfile
      - caddy_data:/data      # 极其关键：Caddy 自动申请的证书、私钥存放于此，遗失会导致高频申请触发限流
      - caddy_config:/config  # 存放 Caddy 运行时的集群状态及临时配置
      - /var/www/html/dist:/var/www/html/dist # 我在此处放了一个 GM 后台代码

volumes:
  caddy_data:
    external: false
  caddy_config:
    external: false

问题一：我使用 vim 修改了 /opt/caddy/Caddfile 之后，使用 sudo docker compose exec -w /etc/caddy caddy caddy reload 重新加载会生效吗？

问题二：我的打包脚本在打包成功之后，为了快，将 /var/www/html/dist 直接删除掉，再将打包好的 dist 目录拷贝到 /var/www/html 目录下面，请问网站还能正常访问吗？

答案是：reload 加载无效、网站无法访问（会返回 404），虽然可以通过重建容器来解决 sudo docker compose down && sudo docker compose up -d ，但是为什么？

我们先来说一下 reload 的问题，我使用 vim 修改了配置之后，发现 reload 不生效，使用 sudo docker compose exec caddy cat /etc/caddy/Caddyfile 查看容器内的配置文件，发现没有生效。

在对问题定位的过程中，AI(Claude/Gemini/GPT) 均想当然的回复说，修改了 Caddyfile 之后，容器内部的配置肯定会生效，并且贴心的给我了一条指令让我验证 echo # test >> Caddyfile。结果当然是没有生效，看不到在文件尾部添加的 “# test”。

但是刚才通过 sudo docker compose up -d 启动之后，立即通过 echo 在 Caddyfile 中追加内容，容器中是可以看到变更的。

原因为何？

通过跟 Claude 的对话中，我发现了问题，因为 echo "# test" >> Caddyfile 之后，是“原位擦写”，而 vim 不同。

先看通过 echo 之后，Caddyfile 的 Inode 是否变更，注意 Inode 未变更，值为 1180275。

其实这个问题早在 2015 年就有人在 Github 提出来1，cpuguy83回复说，因为 mount 是基于 Inode，vim 的修改会创建新文件（先备份原文件，再创建新文件将备份写入，再删除备份文件），会导致了 mount 失效，这是一个已知的文件挂载限制并且不会修复。

那么如何修复这个问题呢？Claude 建议我在 vim 的配置中添加一行 set backupcopy=yes，相当于 echo 原地写入的逻辑，这样在修改文件时，Inode 就不再会发生变更了。

但是我突然有些疑惑，为什么要 vim 要这样实现？我在其他 ubuntu 的机器上熟练的进入 /tmp 目录，使用 echo 创建文件，stat 查看，再通过 vim 修改，发现 Inode 没变化，对你没有看错，没变化！本着不信邪的原则，发现其他机器在 /tmp 目录，通过 vim 修改均不会导致 Inode 变化。

进一步搜索发现，vim 的配置参数对于 backupcopy=auto 在 /tmp 目录的处理机制是其他目录不一致，即在 /opt/caddy 目录下面修改 Caddyfile 时，会创建新文件写入，所以 Inode 会发生变化。但是在 /tmp 却不会。

仔细研究一番之后，因为 /tmp 是一个公共目录，如果采用删除原文件，再写入的方式会引发安全问题。通过 vim 修改时 /tmp/app.conf，若先删除 /tmp/app.conf，则黑客利用这个间隙期，可以执行创建硬链接 ln /etc/passwd /tmp/app.conf，这个写入就会直接将 /tmp/app.conf 写入 /etc/passwd 之中了。如果 root 用户执行一个 chmod 777，则有可能将系统关键配置文件修改成所有人均可修改的。

在 Linux 3.6 内核中2，引入了参数 protected_hardlinks，可以通过 sysctl fs.protected_hardlinks 查看该参数，用于防止 /tmp 以及普通用户可写目录中的 Hard Link TOCTOU（Time Of Check To Time Of Use）提权攻击。这个参数为 1 的情况下，普通用户是无法给类似于 /etc/passwd 的目录在 /tmp 下创建硬链接。

但是否遵循 AI 的建议，将 backupcopy 修改为 yes 呢？ 我的观点是，不要修改，因为 docker-compose.yml 单独挂载一个文件是不合理，而且也不方便。我的解决办法是通过挂载目录来解决，我将 Caddyfile 移动到了 /opt/caddy/config 目录下面。

volumes:
      - ./config:/etc/caddy                  # 挂载整流目录，Vim 产生新 Inode 也在该目录下，容器能正常感知
      - /var/www/html:/var/www/html          # 挂载父级目录，前端打包时在 html 目录下操作，避免 dist 级别的 Inode 丢失

再看另外一个问题，如果删除掉 dist 目录，网站再次访问的时候，为什么 404 了？其实原理仍然是一致的，因为加载的 /var/www/html/dist 的 Inode 发生了变化，原有的 Inode 对应的文件不存在了，新创建的 dist 目录容器内部感知不到，除非重建容器，否则一直访问都会报错 404 Page Not Found。

这虽然是我第一次使用 caddy，但不是我一次使用 docker，事实上我用 docker 快十年了，却在 Inode 上连载两个跟头。通过这次排坑的经验，我发现自己平时还是过于停留在表面了，应该去深入的了解底层原理。虽然有了 AI 的加持，我们的开发速度会快很多，但是真到一些底层的问题上，AI 的建议我们要特别慎重，要多研究一下这些相关参数究竟有什么风险，切勿直接让 AI 接管所有的开发工作。

AI 给出的解决方法往往是解决单一问题的战术方案，而只有深入底层的内核知识，才能帮我们做出兼顾安全与架构的战略决策。
